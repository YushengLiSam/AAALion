## 2. R9.B — Tier 3: social + personalization (#11, #12)

This is where we moved from "a good search box" to "a product with a
point of view."

### #12 Closed-loop preference learning (`addfebe`, `c20f766`)

**Design logic.** We deliberately did **not** train a model. Instead:
- 👍 = +1.0, 👎 = −2.0 (loss-aversion: a reject is a stronger signal than
  a like — matches how people actually shop) to per-(brand/category/
  sub_category) scores, clamped ±10.
- Retrieval applies a **gentle additive prior** *on top of* the rerank
  order: `score = (N − rank) + α·Σ(matching-dim scores)`, α=0.5. Products
  from a 👍'd brand float up; 👎'd ones sink — but relevance still
  dominates, so one tap can't nuke a category.
- **Why additive-on-top, not a hard filter:** explainable (we can show
  the bonus on the "why recommended" card), reversible, and it never
  strands the user with an empty result.
- **Privacy posture (honest framing):** keyed by anonymous per-device id;
  weights never leave our backend; one-tap reset ("我变了"). We say
  "anonymous per-device," NOT "runs on your phone" — retrieval is
  server-side, so the latter would be a lie.

### #11 Group-buy / 拼单 (`6cb42aa`, `9715133`)

**Design logic + the honesty problem.** Real 拼单 needs multiple real
users; we had none. Options were (a) fake it silently, (b) build real
auth first, or (c) **simulate it, labelled**. We chose (c): the modal
shows live progress with simulated 邻居 joining over time, every screen
carries a 演示 disclaimer. The key engineering subtlety:

- **Sticky simulated members.** First implementation derived the
  neighbour count *dynamically* from "headroom" (target − real_count). A
  real join shrank the headroom and **squeezed a simulated neighbour
  back out** — the count got stuck at 2/3. Fix (`56c37f9`): persist
  simulated neighbours as real DB rows via `_ensure_simulated()`, so a
  real action can never evict them. This "materialize the simulation"
  pattern is the right call any time a demo blends real + fake state.

### R9.B bug fixes (`56c37f9`) — all found on the iPhone

1. **Preference reorder didn't apply on re-query.** The response cache
   keyed only on prompt+messages+images, so the 2nd identical query
   replayed the pre-preference order. Fix: fold a sorted-JSON hash of the
   user's weights into the cache key, so any 👍/👎 busts the cache.
   Bumped α 0.3→0.5 so a single tap visibly moves a product.
2. **Group-buy stuck at 2/3** — the sticky-rows fix above.
3. **WeChat share broken** — `ShareLink(item: String)` (WeChat rejects
   bare text). First fix: share a URL. *That introduced bug #4 below.*

## 3. R10 — accounts + the cloud seam

**The realization.** User feedback after the iPhone test: 拼单 is only ever
a simulation because we have no identity — "friends" can't be real
without login. So R10's thesis: **build accounts as the foundation real
multi-user 拼单 (and cross-device preferences) need.**

### Accounts (`9679d65`, `1117195`)

- **Auth methods:** started with Sign in with Apple + 手机号验证码; later
  added **email/phone + password** (see §4 bug 6).
- **The cloud seam — the most important architecture decision of R10.**
  We don't own a cloud user service; Yusheng does (eventually). So the
  backend depends only on a `UserStore` Protocol + a `get_user_store()`
  factory switched by `USER_STORE_BACKEND=local|cloud`. `LocalUserStore`
  = SQLite + mocked SMS for the demo; `CloudUserStore` = a thin HTTP
  proxy to Yusheng's future service. **Flipping one env var moves all
  users to the cloud with zero other code changes.** The 5-endpoint HTTP
  contract is documented so Yusheng can implement against it
  independently. *This is how you let a teammate own a subsystem without
  blocking on them.*
- **Re-key seam on the client:** `DeviceIdentity.userId` returns the
  account id when signed in, else the anonymous IDFV. So **every** feature
  (preferences / group-buy / price-watch / repurchase / chat preference
  prior) re-keys to the account for free, with one-time migration of the
  device's anonymous rows on first sign-in.

### Group-buy polish (`1117195`)
- 拼单成功 → a real **去支付** button (was a dead label) → adds the product
  at the −15% group price → opens the existing CheckoutView.
- Invite shares clean text + an `LP-XXXXX` join code (see §4 bug 5).

### The cloud migration (Yusheng, `c2eb98e` + VM-side)
- Backend moved off Shufeng's Mac to a **GCP VM** (4 vCPU/15 GB),
  systemd-managed, public HTTPS via Cloudflare tunnel. No more "whose Mac
  is on the same WiFi."
- **LLM switched Doubao → TokenRouter haiku.** Doubao's first-token
  latency was ~14 s; haiku ~2 s. Honest tradeoff: we're not on the
  competition's namesake model right now, for a 7× latency win. Quota
  must be watched on demo day.

### Yusheng's parallel work (merged into `6b57d2d`)
- **Retrieval-result cache** (`008238b`): memoizes the expensive,
  preference-*independent* part of `top_k` (hybrid + cross-encoder
  rerank). Repeat query: ~7500 ms → ~1 ms in his measurement. Crucially
  the per-user preference reorder stays **outside** the cache, so #12
  still re-orders live on a cache hit. (Stats helper exists but isn't
  wired into `/cache/stats` yet — cosmetic follow-up.)
- **#7 scene-keyword coverage** + **#8 conversational cart delete**

## 4. R10 bug ledger (problem → root cause → fix)

These are the bugs that actually cost time; preserved so we don't repeat.

1. **Preference cache replay** (§2) — cache key ignored preference state.
   Fix: preference-token in the key.
2. **Group-buy stuck 2/3** (§2) — dynamic simulated count. Fix: sticky rows.
3. **WeChat share = bare String** — WeChat's extension rejects it. Fix:
   share a URL…
4. **…which rendered as raw JSON in WeChat** — the URL pointed at the
   backend's `/groupbuy/{id}` API endpoint, which returns JSON. Fix
   (`46e1e6b`): share **formatted text + an `LP-XXXXX` code**, drop the
   URL entirely (a real Universal Link awaits a stable cloud domain).
5. **Signed-in users got HTTP 400 on 拼单/preferences/price-watch/
   repurchase** — *our own R10 regression.* The route `user_id` regex
   `^[A-Za-z0-9_\-]{8,64}$` rejected the `:` in `phone:…`/`apple:…`. This
   shipped to the cloud before we caught it on the iPhone. Fix
   (`46e1e6b`): widen to `^[A-Za-z0-9_:.@\-]{8,64}$` in all four routes.
   **Lesson:** when you change the *shape* of an identifier, grep every
   validator that touches it.
6. **"SMS code on screen feels fake"** — the demo phone-login shows the
   code on-screen (no real SMS). User wanted something more real. Fix
   (`46e1e6b`): added **email/phone + password** auth (`/auth/register`,
   `/auth/password/login`; pbkdf2_hmac sha256, 100k iters, 16-byte salt).
   First attempt used `hashlib.compare_digest` → 500 (it lives in `hmac`);
   fixed to `hmac.compare_digest`.
7. **👍 highlight reset on leaving a product page** — `prefSignal` was
   transient `@State`. The *server score* was always durable; only the
   button UX was lost. Fix (`46e1e6b`): persist per (userId, productId)
   in UserDefaults, restore on view appear.
8. **Cloud served stale code** — Yusheng deployed via tarball, so `git
   pull` on the VM didn't apply our `46e1e6b`. Caught it by probing the
   live cloud (`/auth/register` → 404). Fix: Yusheng repackaged →
   `6b57d2d` live; we recommended converting the VM to a git-clone +
   read-only deploy key so future redeploys are one command.

**Cross-cutting operational pain:** the Cloudflare *quick* tunnel URL
rotates on every restart (~4 rotations these rounds), each forcing a
Config.swift rebake + rebuild. The auto-purge logic (`2b77e57`) made this
survivable. A stable domain is on the cloud roadmap.

## 5. Design logic, distilled

Three principles emerged that explain most of our choices:

1. **Make the reasoning visible.** Hallucination receipts, "why
   recommended" cards, per-source citations, the 演示 disclaimers — every
   one trades a little screen space for trust. This is deliberate: a
   judge can't grade what they can't see.
2. **Simulate honestly, materialize the simulation.** Where we can't have
   the real thing yet (multi-user 拼单), we fake it *labelled* and persist
   the fake state as real rows so it behaves consistently. Never a silent
   fake.
3. **Seam, don't couple.** The `UserStore` cloud seam, the `LLM_PROVIDER`
   switch, the `DeviceIdentity` re-key — each lets a piece (cloud user
   service, model, identity) change behind a stable interface without a
   rewrite. This is what let three people work in parallel and merge
   cleanly into `6b57d2d`.
